1//! # metw-accounts-center service
2//!
3//! This crate is the abstract orchestrating microservice for metw.cc. It
4//! manages clients and data repositories, and provides a library interface to
5//! use with a presentation layer.
6//!
7//!
8//! ## System Design
9//!
10//! This project strictly adheres to the [Separation of Concerns][SoC] (SoC)
11//! principle. The service abstraction layers are:
12//! - [`repo`]: Defines traits for application storage, database, and cache
13//!   access.
14//! - [`service`]: Provides an interface for manipulating application state.
15//!   Elaborates on low-level transactions defined in the `repo` layer to the
16//!   upper layers.
17//! - [`handlers`]: The final level of abstraction to manipulate the
18//!   application state. This layer is a transport-agnostic front end for
19//!   the microservice.
20//! - [`AppState`]: The application state contains the services constructed
21//!   from repositories and clients. Handlers require a state instance to
22//!   construct.
23//!
24//! ### Supporting Modules
25//!
26//! [`client`] is used for external integrations (e.g., email provider,
27//! CAPTCHA).
28//!
29//! The [`dto`] module defines the interface to communicate with `handlers`.
30//! [`dto::service`], [`dto::repo`] and [`entity`] are used for internal
31//! communication between `service` and `repo` layers.
32//!
33//! [`util`] is used for miscellaneous utilities that do not fit into
34//! categories above. Check out documentations of the `util` for details.
35//! Conceptually, the [`token`] and [`id`] modules are re-exported at the crate
36//! root as they are too common.
37//!
38//! [SoC]: https://en.wikipedia.org/wiki/Separation_of_concerns
39
