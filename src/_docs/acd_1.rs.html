1//! Take a look at the [System Design](service#system-design) section in the
2//! service crate.
3//!
4//! This documentation discusses the communication between the `repo`, `service`,
5//! and `handler` layers.
6//!
7//!
8//! ## `repo`
9//!
10//! The [`repo`] layer is responsible for data management.
11//!
12//! Methods of `*Repo` traits provide read-only access or test-and-set
13//! operations. `*RepoTransaction` traits, on the other hand, are atomic
14//! transactional units to manipulate data freely. For example, [`AccountRepo`]
15//! or its transaction variant [`AccountRepoTransaction`]. Implementation
16//! conventions for this layer:
17//!
18//! - Non-transaction repo operations can *only be* read or atomic
19//!   compare-and-swap / test-and-set.
20//!
21//! | Function Example | Type | Description |
22//! |--|--|--|
23//! | [`get_keys`] | read | Reads keying materials of users. |
24//! | [`set_primary_email_if_current_is`] | compare-and-swap | If the provided primary email is still the current primary email, changes the email. |
25//!
26//! - Any insert, delete, update, or upsert (insert or update if exists)
27//!   operation must reside within a transaction.
28//!
29//! | Function Example | Type | Description |
30//! |--|--|--|
31//! | [`upsert_account`] | upsert | Inserts a new user or updates the existing one by account id. |
32//! | [`insert_default_flags`] | insert | Inserts default user flags. |
33//!
34//! - `*Repo` variants that return a single item like "fetch one" do not return
35//!   a `RowNotFound` error; they must be mapped to an `Option`. Those that
36//!   are "fetch many" return a `Vec`.
37//!
38//! | Function Example | Type | Description |
39//! |--|--|--|
40//! | [`get_primary_email`] | fetch one | Returns the primary email of the user. |
41//! | [`get_secondary_emails`] | fetch many | Returns the secondary emails of the user. |
42//!
43//! - [`RepoError`] is an internal error used only for tracing and
44//!   observability. This error must not be returned directly to the user; it
45//!   must be *redacted*.
46//!

62//! ## `service`
63//!
64//! The [`service`] layer acts as the "middleman" between the handlers and the
65//! data access (repo) layers. It abstracts low-level access operations into
66//! units of work.
67//!
68//! - High-level business logic validation errors are mapped to specific
69//!   variants of [`ServiceError`]. Low-level database infrastructure failures
70//!   bypass specific mappings and propagate as generic errors.
71//!
72//! As an example, error conditions in [`AccountService::signup`] are:
73//! | Condition | Error Type |
74//! |--|--|
75//! | Username is already taken | [`ServiceError::UsernameTaken`] |
76//! | Email is already taken | [`ServiceError::EmailTaken`] |
77//!
78//! The service layer handles specific error mapping on a *best-effort* basis
79//! by validating conditions before executing raw changes. If a validation
80//! fails to predict a state conflict, or if an underlying database constraint
81//! triggers an unhandled error, the fallback result will be a generic
82//! [`ServiceError::Repo`]. In such scenarios, upper layers must continue to
83//! treat the internal transparent errors as redacted.
84//!
85//! [`service`]: service::service
86//!
87//! [`AccountService::signup`]: service::service::AccountService::signup
88//!
89//! [`ServiceError`]: service::service::ServiceError
90//! [`ServiceError::UsernameTaken`]: service::service::ServiceError::UsernameTaken
91//! [`ServiceError::EmailTaken`]: service::service::ServiceError::EmailTaken
92//! [`ServiceError::Repo`]: service::service::ServiceError::Repo

95//! ## `handlers`
96//!
97//! The [`handlers`] layer composes multiple services to orchestrate complex,
98//! unit-of-works. Generally, the handler layer executes one or more
99//! service calls and either propagates the service results directly or
100//! translates them into specific handler errors based on the business outcome.
101//!
102//! As an example, error conditions in [`AuthenticationHandler::auth_session`] are:
103//!
104//! | Condition | Error Type |
105//! |--|--|
106//! | Token scope is not [`TokenScope::Session`] | [`HandlerError::Unauthorized`] |
107//! | JWT-related issue (triggered from TokenService) | [`HandlerError::Service`] |
108//!
109//! - There are two types of handlers: those for HTTP endpoints and those for
110//!   middleware.
111//!
112//! - In handlers that are HTTP endpoints, at first request validation happens.
113//!   Everything coming from the user *must be* a [`dto::request`]; and the
114//!   handler response, if exists, must be [`dto::response`].
115//!
116//! - Middleware handlers may return types other than [`dto::response`].
117//!
118//! - The handler owns the allocation and passes references to the service
119//!   layer. It, however, transfer ownership of the allocation to `client`s.
120//!
121//! [`handlers`]: service::handlers
122//!
123//! [`ServiceError`]: service::service::ServiceError
